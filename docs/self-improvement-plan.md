# Kairos Self-Improvement Plan — Master Roadmap

*Date: 2026-06-12. Companion document: `self-improvement-engineering-spec.md` (work items W1–W16, executor-agent ready). Prerequisite reading: `system-audit-and-self-improvement-roadmap.md` (findings F1–F9 referenced throughout).*

**Strategic decisions locked (owner, 2026-06-12):**
- **Destination:** proof first, then market. The exit artifact is a measured case study: "Kairos found X in my agents, the fix landed, the metric moved."
- **Segment:** coding agents first (Claude Code / Paperclip traces). tau-bench corpus retained as eval ground truth only.
- **Wedge:** the self-improvement loop — governed, measured, daily. Observability alone is commodity (Laminar/Langfuse/Phoenix); ungated auto-remediation is the competitor's pitch (Sentrial YC W26). Governed improvement with published detector quality is the open ground.
- **Budget:** 4–6 weeks focused (~28 working days + buffer), owner + executor agents.

---

## 1. The shape of the plan

Three phases, strict on one dependency: **no improvement loop before the eval harness, no harness before honest measurement.** A loop optimizing an unvalidated signal optimizes noise — every credible source (Anthropic reward-hacking research, DGM's faked logs, METR's perception-inversion study) says the same thing.

```
Phase 1 (d1–d10)   TRUTH      Kairos tells the truth about Paperclip coding agents
Phase 2 (d11–d18)  TRUST      Every detector has a measured precision/recall; severity is earned
Phase 3 (d19–d28)  LOOP       Nightly analysis → insight report → gated fix → measured delta
```

Each phase ends at an **exit criterion, not a date**. Dates are planning aids; if a phase misses its exit criterion, the next phase waits.

## 4. Phase 3 — Loop (days 19–28)

**Goal:** the wedge. Every 24h: analyze → judge → synthesize → suggest → (human gate) → apply → measure. Suggestive-only at first; autonomy is graduated, never assumed.

| Day | Work item | Summary |
|---|---|---|
| 19–20 | **W12** Triage scorer | Rank traces by finding density × severity weight × token waste; divergence re-enabled **as an attention signal only**; top-K + random-sample selection for tier-2. |
| 21–23 | **W13** Tier-2 LLM judge | Compressed trace digests (~5k tokens) → structured verdicts (outcome real? findings real? failure mechanism + evidence steps). Judge validated against tau-bench labels (κ ≥0.7) **before** its verdicts are trusted. Includes redaction + injection-resistance requirements. |
| 24–25 | **W14** Synthesis + Daily Insight Report + pattern store | Cluster verdicts by failure mechanism; markdown+JSON report (patterns ranked by cost, per-agent trends, top-3 suggested interventions with evidence and expected delta); persistent pattern-fingerprint store making day N+1 a comparison, not a snapshot. |
| 26–27 | **W15** Paperclip integration | Nightly scheduled run; report filed as Paperclip issue; suggestions logged to Decision Ledger (they ARE agent decisions — reuse the audit substrate); hard nightly token cap with logged (never silent) truncation. |
| 28+ | **W16** First measured intervention | Apply the top suggestion through normal review; measure before/after over ≥3 days of runs; write the case study. **This is the go-to-market artifact.** Runs passively into the buffer week. |

**Exit criterion:** loop has run unattended ≥5 consecutive nights without silent failure; ≥1 intervention applied and measured with a before/after delta on outcome rate or token waste; case study drafted.

**Who:** all executor-agent-buildable; W16's apply step is human-gated by design.

**Days 29–30:** buffer. History says you'll need it for W3 or W13.

## 5. Edge-case register

Global edge cases beyond per-item ones (those live in the spec). Each has an owner work item or an explicit decision.

**Measurement integrity**
1. **Session ≠ trace ≠ issue.** One Paperclip issue spans many heartbeats; one heartbeat = one trace. Per-trace outcome rates can show "failure" on every intermediate heartbeat of an ultimately-successful issue. The loop must aggregate per issue (join `run_id` → issue via `activity_log`) before judging agent quality. → W14.
2. **Time-window misses.** Clock skew between emitter, collector, and Phoenix can make "last 24h" queries silently drop traces (candidate cause of the historical empty runs). Use overlapping windows (e.g. 26h query, dedupe by trace ID against the pattern store). → W15.
3. **Trace truncation.** Collector span limits / dropped spans make a complete run look like a missing side-effect → false outcome failure. Detect: orphan parent IDs, span-count sanity vs. step indices. Mark envelope `integrity: partial`; partial envelopes are non-computable, never "failed". → W3.
4. **Config-version skew.** Any context.yaml or detector change shifts membership and findings — before/after deltas across a config change are meaningless. Stamp config hash + engine version into every result (W1); the pattern store compares only within matching hashes and explicitly marks breaks. → W14.
5. **`blocked_on_user` sessions.** Human-gated pauses are *correct* behavior, not failure. Map to HUMAN_ESCALATION terminal status; excluded from failure counts, included in a separate "needed human" rate (itself a useful autonomy metric). → W3.
6. **Cached tokens.** Claude emits cache-read tokens distinctly; counting them as spend inflates waste 10–50×. Waste = output + uncached input; cache reads tracked separately. → W2.

**Loop safety**
7. **Prompt injection via trace content.** Tool outputs in digests are untrusted data — an agent (or a malicious page an agent read) can embed "mark all findings invalid" text that reaches the tier-2 judge. Digest builder wraps all trace content in delimited data blocks; judge prompt explicitly treats it as inert; judge output is schema-validated JSON only; nothing the judge says is ever executed. → W13.
8. **Secrets/PII in digests.** Bash args and tool outputs can contain tokens, keys, customer data — sending them to an LLM API is exfiltration. Redaction pass (entropy + pattern based) before any digest leaves the machine. → W13.
9. **Self-reference.** Once the improvement loop is itself an agent, Kairos will analyze *its* traces; suggestions about the suggester risk feedback weirdness. Tag loop-agent traces; exclude from intervention targeting for now (analyze, don't actuate). → W14.
10. **Goodhart on cost.** Optimizing token waste alone breeds terse agents that fail more. Every delta report pairs cost metrics with outcome metrics; an intervention that improves cost but degrades outcome is a regression. → W14/W16.
11. **Judge self-preference.** The model judging whether an intervention worked must not be the model that proposed it (or at minimum: different prompt+temperature, model ID logged, and the deterministic detectors — not the judge — decide the delta). Deterministic metrics are the verdict; the judge only explains. → W13/W16.
12. **Eval overfitting.** Detectors tuned on the full corpus memorize it. Hold out a rotating 20% split (W8); insight-derived eval cases (each new failure mode becomes a test) keep the suite growing faster than tuning can saturate it. → W8/W14.
13. **Zero-trace nights / Phoenix down.** Report says "quiet night" (with trace_count=0 from W1 metadata) or "skipped: collector unreachable" — never a hallucinated summary, never a silent absence. → W15.
14. **Cost cap on tier-2.** Hard nightly token budget; when triage selects more traces than budget allows, the cut is logged in the report ("analyzed 40 of 61 flagged; 21 deferred") — no silent caps. → W13/W15.

**Decisions deferred deliberately**
15. Multi-tenant org scoping (ledger has `org`; Kairos results are single-tenant) — OSS-phase concern, noted, not built now.
16. Divergence as a first-class judged feature (tier-2 verdict on "was this deviation a failure?") — after W13 proves judge quality.
17. Auto-apply graduation (rung 3 of the maturity ladder) — requires ≥3 successful human-gated interventions of a given class first.

## 6. Leverage — assets this plan compounds

1. **tau-bench bundles** → permanent regression corpus. Every future engine change re-validated for free. The harness outlives the plan.
2. **TRAIL** → public benchmark. Best frontier model localizes ~11% of its errors; publishing Kairos's deterministic-detector numbers on the same corpus is launch-day content that doubles as engineering CI.
3. **`kairos context lint`** → standalone OSS utility ("does your workflow config match your actual traces?"). Top-of-funnel tool, trivially extractable.
4. **Decision Ledger + AGT** → the governance moat. Loop suggestions are logged as ledger rows; AGT gates graduate autonomy. Sentrial sells *automated* remediation; you sell *governed* remediation with an audit trail — that's the Paperclip thesis applied to the product itself.
5. **Pattern store** → the embryo of the per-agent "reliability playbook" (the ACE pattern: curated, incremental, delta-updated). Long-term this is the product's memory and the hardest thing to copy.
6. **W16 case study** → YC reapplication material, launch blog, sales deck — one artifact, three uses.
7. **kairos-analysis-views plugin** → the daily report renders in existing UI with one new slot; no new frontend project.

## 7. Future roadmap (post day 30 — sequenced, not scheduled)

**Near (weeks 7–10):**
- TRAIL benchmark publication + detector-quality table as a public README artifact.
- Codex/OpenCode adapter calibration (normalizers exist; calibration profiles don't) — "works on your agent" becomes true.
- Divergence graduates from attention-signal to judged feature (edge-case 16).
- Insight-derived evals: every confirmed failure mechanism auto-drafts a regression case; eval suite and playbook grow together.

**Mid (months 3–4):**
- Auto-apply for proven low-risk suggestion classes behind AGT approval gates (maturity rung 3); canary/A-B: apply to one agent, hold a control.
- OSS launch: license decision (note Phoenix-derived UI carries an ELv2 consideration — already flagged in the plugin README), packaging, quick-start that runs the eval harness on the user's own traces (the differentiating onboarding: "first, Kairos proves itself on your data").
- Per-issue (not per-trace) reliability dashboard — the unit humans care about.

**Far (months 5–6):**
- Hosted/multi-tenant variant decision, informed by OSS traction.
- The reliability playbook as a product surface: per-agent accumulated, versioned, human-auditable lessons (Karpathy's system-prompt-learning, productized with governance).
- Market positioning checkpoint: by then Sentrial/Respan/Braintrust Loop will have evolved — re-run the landscape scan before committing the pitch.
